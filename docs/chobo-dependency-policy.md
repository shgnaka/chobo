# CHOBO 依存採用ポリシー

## 1. 目的

この文書は、パッケージ・プラグイン・ライブラリを採用する前に、どの観点で比較し、どの条件なら自前実装を優先するかを決めるための判断基準を定める。

## 2. 基本方針

- 依存は増やせるが、増やすたびに保守対象と障害面積も増える
- まずは自前実装と既存依存の再利用を比較する
- 安易な採用より、必要性が説明できる依存だけを入れる
- 重要機能ほど、依存の更新停止や破綻時の代替手段を考える

## 3. 採用前に確認すること

次の問いにすべて答えられるか確認する。

1. 何を解決したいのか
- 採用候補が解決する課題を 1 文で言えるか

2. 自前実装と比べて何がよくなるのか
- 実装速度
- 保守負担
- 安全性
- 一貫性
- iOS / Android 両対応のしやすさ

3. 自前実装した場合の負担はどれくらいか
- 実装工数
- 将来の変更容易性
- OS 差分吸収の手間

4. 依存の保守は誰が追うのか
- 更新頻度
- 破壊的変更
- セキュリティ修正の追従

5. 代替手段はあるか
- OS 標準 API
- 既にある社内実装
- もっと小さい依存

6. その依存を外せるか
- 将来の置き換え方法
- 抽象化境界
- 依存を閉じ込める場所

## 4. 判断フロー

### 4.1 まず分類する

次のどれかを決める。

- `must` : その依存がないと要件を満たせない
- `should` : 採用すると大きく楽になる
- `can avoid` : 自前実装でも十分いける

### 4.2 原則

- `can avoid` は原則として採用しない
- `should` は比較表を作ってから採用する
- `must` は採用するが、代替と退出条件を必ず書く

### 4.3 採用判断

採用してよい条件は次の通り。

- CHOBO の目的に直接効く
- 既存実装より総コストが低い
- Android と iOS の両方で破綻しにくい
- メンテナンス対象として現実的
- 依存更新で基盤が壊れにくい

## 5. 自前実装を優先する条件

次のどれかに当てはまるなら、自前実装を強く優先する。

- 要件が小さく、実装が単純
- 依存を入れるほうが複雑になる
- セキュリティや秘密情報の扱いが重い
- OS ごとの差分が少なく、薄いラッパーで足りる
- 依存の更新頻度が高すぎる
- 依存の責務が広すぎる

## 6. 依存採用の記録

依存を採用するときは、短いメモを残す。

- 採用理由
- 解決したい課題
- 自前実装しない理由
- 対象プラットフォーム
- 更新責任者
- 代替案
- 将来の撤去方法

## 7. CHOBO での運用

CHOBO では次の順で判断する。

1. まずドメイン層と保存ルールを自前で固める
2. その上で OS 固有機能だけを最小限に依存する
3. UI は共通コードベースを優先する
4. 依存は増やしすぎない
5. 追加前に必ずこの文書で確認する

Flutter での候補仕分けは [`docs/chobo-flutter-dependency-matrix.md`](/home/shgnaka/chobo/docs/chobo-flutter-dependency-matrix.md) にまとめる。

## 8. 追加時の合格ライン

以下を満たさない依存は入れない。

- 目的が明確
- 代替案比較が済んでいる
- 影響範囲が理解できている
- iOS / Android の両方で成立する見通しがある
- テスト方針がある
- バージョン固定方針がある
- 破綻時の撤退方針がある
