## Problem

一般的な家計簿アプリは、支出カテゴリの集計には強いが、「給与が入ってから固定費・貯蓄・投資・日常支出へどう流れたか」「クレジットカード利用がどの時点で負債になり、引き落としでどう解消されたか」のような資金移動の連鎖を捉えにくい。  
その結果、月次の合計額は見えても、手元資金、未払債務、積立、資産移動の関係が頭の中で分断されやすい。  
家計管理で本当に知りたいのは費目別合計だけではなく、家計全体の中でお金がどの勘定からどの勘定へ移り、どの判断が将来のキャッシュポジションを変えたかである。

## Idea

家計簿アプリの中核に、複式簿記をそのまま全面採用するのではなく、家計向けに簡略化した「勘定ベースの資金フロープロトコル」を置く。  
各取引は少なくとも「どこから減ったか」と「どこへ増えたか」を持ち、アプリはその記録から残高推移と資金移動の流れを再構成する。

最小モデルでは勘定を次の4群に分ける。

1. 資産
- 現金、普通預金、電子マネー、証券口座、積立口座など

2. 負債
- クレジットカード未払、立替、ローンなど

3. 収益
- 給与、副収入、還付、ポイント還元など

4. 費用
- 食費、住居費、通信費、娯楽費、教育費など

このモデルで、家計内イベントをすべて「勘定間の移動」として表す。

- 給与受取: `収益 -> 普通預金`
- 現金引き出し: `普通預金 -> 現金`
- スーパーで現金支払い: `現金 -> 費用:食費`
- クレジットカードで買い物: `負債:カード未払 -> 費用:食費` ではなく、家計向けに `費用:食費` の発生と `負債:カード未払` の増加を同時に扱うテンプレートとして記録する
- カード引き落とし: `普通預金 -> 負債:カード未払`
- 貯蓄口座へ移動: `普通預金 -> 資産:積立口座`

ユーザー向けUIは会計用語を前面に出しすぎず、裏側のプロトコルだけを会計的にする。

- 入力時は「収入」「支出」「移動」「カード払い」などの生活用語で選べる
- 内部では勘定科目と仕訳テンプレートに変換する
- 表示ではカテゴリ集計に加えて「今月の資金フロー図」や「口座別残高の変化」を出す

この方式にすると、通常の家計簿体験を保ちつつ、以下の問いに答えやすくなる。

- 今月の支出が増えた原因は、どの費用勘定か
- 可処分資金はどの資産勘定に滞留しているか
- クレジット利用が将来キャッシュをどれだけ圧迫するか
- 貯蓄や投資への振替は実際にどれだけ実行できているか

初期MVPでは、分析テーマを「お金の流れの可視化」に絞る。  
予算最適化、家族共有、レシートOCR、自動連携などは後回しにし、まずは勘定ベースの記録で本当に理解しやすくなるかを検証する。

## Protocol Spec

### 1. Goal

資金フロープロトコルは、家計内のあらゆる取引を「勘定の増減イベント」として表現し、次の3つを一貫して再構成できることを目的とする。

- 現時点の勘定残高
- 一定期間におけるお金の流入・流出・移動
- 将来キャッシュに影響する未払負債の残高

このプロトコルは、UI入力形式とは切り離された内部表現である。  
ユーザーは生活用語で入力し、アプリ内部でプロトコル準拠の取引データへ変換する。

### 2. Core Concepts

#### 2.1 Account

`Account` は残高または累計を持つ最小単位の勘定で、以下の属性を持つ。

- `account_id`: 一意識別子
- `name`: 表示名
- `kind`: `asset | liability | income | expense | equity`
- `parent_account_id`: 任意。階層化用
- `currency`: 初期MVPでは単一通貨前提
- `is_archived`: 利用停止フラグ

`equity` は会計上の純資産に相当するが、家計UIでは直接見せない内部勘定として使う。  
初期MVPでは `income` と `expense` の期間損益を `equity` に閉じる厳密処理は後回しにし、可視化上は `asset` と `liability` の差分を純資産相当として扱ってよい。

#### 2.2 Entry

`Entry` は1つの勘定に対する増減を表す最小記録単位で、以下を持つ。

- `account_id`
- `direction`: `increase | decrease`
- `amount`: 正の金額
- `memo`: 任意

#### 2.3 Transaction

`Transaction` は複数の `Entry` を束ねたイベントで、家計上の1出来事を表す。

- `transaction_id`
- `date`
- `type`
- `status`: `posted | pending | void`
- `entries`
- `counterparty`: 任意
- `description`: 任意
- `external_ref`: 任意。レシートIDやインポート元参照

1つの `Transaction` は、必ず2件以上の `Entry` を持つ。

### 3. Account Kinds and Balance Semantics

勘定種別ごとの増減の意味を固定する。

| kind | `increase` の意味 | `decrease` の意味 |
| --- | --- | --- |
| `asset` | 資産が増える | 資産が減る |
| `liability` | 負債が増える | 負債が減る |
| `income` | 収益が増える | 収益を打ち消す |
| `expense` | 費用が増える | 費用を打ち消す |
| `equity` | 純資産が増える | 純資産が減る |

この定義により、プロトコル上では「借方 / 貸方」を直接露出しなくても、勘定種別と増減方向だけで処理できるようにする。

### 4. Transaction Types

初期MVPの `type` は次の5種類とする。

1. `income`
- 外部から家計へ資金が入る取引
- 例: 給与受取、還付、副収入

2. `expense`
- 家計資産を使って費用が発生する取引
- 例: 現金払い、デビット払い、口座引落の即時支出

3. `transfer`
- 家計内の勘定間で資金を移す取引
- 例: 普通預金から現金へ移動、預金から積立口座へ振替

4. `credit_expense`
- 費用は確定するが、現時点では現金が出ず、負債が増える取引
- 例: クレジットカード利用、後払い決済

5. `liability_payment`
- 既存負債を資産から返済する取引
- 例: カード引き落とし、ローン返済

### 5. Canonical Transaction Shapes

各 `type` は以下の標準形へ正規化する。

#### 5.1 Income

- `income` account: `increase`
- `asset` account: `increase`

例: 給与受取

- `income:salary` `increase` 300000
- `asset:bank:main` `increase` 300000

#### 5.2 Expense

- `asset` account: `decrease`
- `expense` account: `increase`

例: 現金で食費支払い

- `asset:cash` `decrease` 2500
- `expense:food` `increase` 2500

#### 5.3 Transfer

- 送金元 `asset`: `decrease`
- 送金先 `asset`: `increase`

例: 普通預金から積立口座へ振替

- `asset:bank:main` `decrease` 50000
- `asset:saving` `increase` 50000

`transfer` は純支出ではないため、`expense` を増やしてはならない。

#### 5.4 Credit Expense

- `liability` account: `increase`
- `expense` account: `increase`

例: クレジットカードで食費支払い

- `liability:card:visa` `increase` 4200
- `expense:food` `increase` 4200

この時点では `asset` を減らさない。  
キャッシュ流出は後続の `liability_payment` で表現する。

#### 5.5 Liability Payment

- `asset` account: `decrease`
- `liability` account: `decrease`

例: 銀行口座からカード引き落とし

- `asset:bank:main` `decrease` 4200
- `liability:card:visa` `decrease` 4200

### 6. Posting Rules

プロトコルに投入される `Transaction` は、以下の整合性条件を満たす必要がある。

1. 金額はすべて正数
- `Entry.amount <= 0` は不正

2. `entries` は2件以上
- 単独エントリの取引は許可しない

3. 取引総額は釣り合う
- 厳密な借貸表現の代わりに、標準形に従う `type` ごとの必須勘定組み合わせを満たすことをもって整合性とする

4. `transfer` は同一通貨の `asset` 間のみ
- 初期MVPでは為替差損益は扱わない

5. `credit_expense` は必ず `liability` を含む
- 負債勘定がなければ後日の返済追跡ができない

6. `liability_payment` は必ず既存負債を減らす
- 返済後に負債残高が負になる入力は不正

7. `void` 取引は残高計算対象外
- 誤入力取消は削除ではなく `status = void` を優先する

### 7. Balance Update Rules

残高は、時系列順に `posted` 取引だけを適用して算出する。

- `asset` 残高 = `increase` 合計 - `decrease` 合計
- `liability` 残高 = `increase` 合計 - `decrease` 合計
- `income` 累計 = `increase` 合計 - `decrease` 合計
- `expense` 累計 = `increase` 合計 - `decrease` 合計

月次可視化では、以下を主要指標として出す。

- `net_assets = total_assets - total_liabilities`
- `cash_out_expenses = 現金または預金減少を伴った expense の合計`
- `accrued_expenses = credit_expense により発生した未払費用の合計`
- `liability_due = 未返済 liability 残高`

### 8. Credit Card Rules

クレジットカードは「利用時の費用認識」と「引落時の資産減少」を分離して扱う。

#### 8.1 Card Purchase

- `type = credit_expense`
- 発生日に費用を認識する
- 同時にカード未払を増やす

#### 8.2 Card Payment

- `type = liability_payment`
- 引落日に預金を減らす
- 同時にカード未払を減らす

#### 8.3 Installment or Split Payment

初期MVPでは分割払いを厳密には扱わず、次の簡略仕様とする。

- 利用時に総額を `liability` へ計上する
- 分割手数料は別 `expense` として月次で追加記録する
- 分割回数や将来支払予定表は v2 以降で管理する

### 9. UI to Protocol Mapping

UI入力は以下のように内部取引へ変換する。

| UI入力種別 | 内部 `type` | 必須選択項目 |
| --- | --- | --- |
| 収入 | `income` | 入金先資産勘定、収益カテゴリ |
| 支出 | `expense` | 支払元資産勘定、費用カテゴリ |
| 口座移動 | `transfer` | 送金元資産勘定、送金先資産勘定 |
| カード払い | `credit_expense` | 負債勘定、費用カテゴリ |
| カード支払 | `liability_payment` | 支払元資産勘定、対象負債勘定 |

この変換層により、ユーザーは複式簿記を意識せずに、内部では整合した取引データを保持できる。

### 11. Non-Goals for MVP

- 複数通貨対応
- 厳密な会計期間締め処理
- 減価償却や税計算
- 家族共有時の個人別持分管理
- 銀行APIやカード明細APIとの自動同期

### 12. Household Event Mapping Table

代表的な家計イベントを、初期MVPの5取引型へマッピングする。

| # | 家計イベント | `type` | 標準エントリ | メモ |
| --- | --- | --- | --- | --- |
| 1 | 給与が普通預金に振り込まれる | `income` | `income:salary increase` / `asset:bank:main increase` | 定期収入の基本形 |
| 2 | 副業報酬が普通預金に入る | `income` | `income:side_job increase` / `asset:bank:main increase` | 収益カテゴリだけ差し替える |
| 3 | 現金で食材を買う | `expense` | `asset:cash decrease` / `expense:food increase` | 即時支出 |
| 4 | デビットカードで日用品を買う | `expense` | `asset:bank:main decrease` / `expense:daily_goods increase` | 銀行残高が即時減る支出 |
| 5 | 口座引落で家賃を払う | `expense` | `asset:bank:main decrease` / `expense:rent increase` | 固定費の即時支出 |
| 6 | 現金をATMで引き出す | `transfer` | `asset:bank:main decrease` / `asset:cash increase` | 支出ではなく資産移動 |
| 7 | 現金を銀行に預け入れる | `transfer` | `asset:cash decrease` / `asset:bank:main increase` | 逆向きの資産移動 |
| 8 | 普通預金から積立口座へ移す | `transfer` | `asset:bank:main decrease` / `asset:saving increase` | 貯蓄の実行量を追える |
| 9 | 普通預金から証券口座へ入金する | `transfer` | `asset:bank:main decrease` / `asset:brokerage increase` | 投資原資の移動 |
| 10 | クレジットカードで食費を払う | `credit_expense` | `liability:card:main increase` / `expense:food increase` | 費用と負債を同時認識 |
| 11 | クレジットカードで電車定期を買う | `credit_expense` | `liability:card:main increase` / `expense:transport increase` | カード支出の別カテゴリ例 |
| 12 | QR後払いで雑貨を買う | `credit_expense` | `liability:bnpl increase` / `expense:daily_goods increase` | 後払い系も同型 |
| 13 | カード利用分が普通預金から引き落とされる | `liability_payment` | `asset:bank:main decrease` / `liability:card:main decrease` | 利用時の費用認識とは分離 |
| 14 | ローン返済が口座から引き落とされる | `liability_payment` | `asset:bank:main decrease` / `liability:loan decrease` | 元本返済部分の基本形 |
| 15 | 家族の立替分を現金で精算する | `expense` | `asset:cash decrease` / `expense:family_support increase` | 家計外への支出として扱う簡略形 |
| 16 | 友人に立て替えてもらった食事代を送金で返す | `expense` | `asset:bank:main decrease` / `expense:social increase` | 初期MVPでは未払管理を省略した簡略形 |
| 17 | ポイント還元が電子マネー残高へ付与される | `income` | `income:point_reward increase` / `asset:e_money increase` | 家計への価値流入として扱う |
| 18 | 税金の還付金が口座に入る | `income` | `income:tax_refund increase` / `asset:bank:main increase` | 臨時収入の例 |
| 19 | 電子マネーへ銀行口座からチャージする | `transfer` | `asset:bank:main decrease` / `asset:e_money increase` | 消費前のチャージは支出ではない |
| 20 | 電子マネー残高でコンビニ支払いをする | `expense` | `asset:e_money decrease` / `expense:food increase` | チャージ時ではなく利用時に費用計上 |

この表から、初期MVPで特に重要な判断は次の3点である。

- `transfer` を支出と混同しないこと
- `credit_expense` と `liability_payment` を必ず別取引として記録すること
- チャージや貯蓄移動のような資産内移動は、可処分資金の所在変化として扱うこと

一方で、次のイベントは追加ルールが必要になりやすい。

- 分割払い、リボ払い、利息付きローン返済
- 立替金を資産・負債として厳密管理したいケース
- 返品、返金、支出取消のような逆仕訳イベント
- 資産売却益や配当のように `income` と `asset` の複合解釈が必要な投資イベント

これらの例外イベントは、初期MVPでは構想外として扱い、まずは通常的な家計イベントを安定して記録・可視化できることを優先する。

## 検討事項

### 1. 利用環境の前提

- 個人利用のローカルアプリとして始めるか
- スマホ中心で使う前提にするか、PC中心で使う前提にするか
- Webアプリ、モバイルアプリ、ローカルデスクトップアプリのどれを初期提供形態にするか
- オフライン利用を前提にするか、常時オンライン前提にするか
- 単一端末で完結させるか、複数端末同期を初期から考えるか
- 毎日の入力を重視するか、週次・月次の振り返りを重視するか

### 2. データ保管方式

- 完全ローカル保存にするか
- ローカル保存を基本にしつつ、任意バックアップだけクラウドに置くか
- クラウド保存を前提にして複数端末同期を優先するか
- 保存単位を `勘定マスタ / 取引履歴 / 月次集計キャッシュ / 設定` に分けるか
- 元データをイベントソーシング的に `Transaction` の追記ログとして持つか
- 残高や月次集計は都度再計算するか、キャッシュテーブルを持つか
- エクスポート形式を JSON, CSV, Markdown のどれで用意するか
- インポート・エクスポート時にプロトコルの完全情報を保持するか、閲覧用の簡略形式も別に持つか

### 3. 利用環境と保管方式の組み合わせ案

- ローカルファーストなスマホアプリ
- ローカルDBに保存し、端末ロックとOSの安全領域に依存できる
- 入力のしやすさは高いが、複雑な可視化や分析画面は作りにくい可能性がある

- ローカルファーストなデスクトップアプリ
- ファイルベースまたは組み込みDBで持ちやすく、分析UIやエクスポートも作りやすい
- 日常入力の即時性はスマホに劣る可能性がある

- PWA / Webアプリ
- 配布しやすく、将来の同期機能へもつなげやすい
- ブラウザ保存領域の制約や秘密情報保護の設計がやや難しい

- ローカル保存 + 暗号化バックアップ
- 普段は端末内だけで扱い、バックアップ時だけ暗号化ファイルを書き出す
- MVPでの安全性と実装負荷のバランスが比較的よい

### 4. セキュリティ上の基本方針

- 金融データを含むため、初期から「漏えい前提で困る情報」として扱う
- MVPでも平文での外部送信は避ける
- 不要な個人情報は持たない
- 金額、勘定名、メモ、店舗名など、どこまで機微情報とみなすかを定義する
- 「便利さ優先のクラウド同期」と「安全性優先のローカル保持」のどちらを優先するか方針を決める

### 5. 保存時の保護

- 端末内保存データを暗号化するか
- アプリ独自パスフレーズで暗号化するか、OSのキーチェーン/安全領域を使うか
- SQLite などのDBを使う場合、DB全体暗号化を行うか
- ファイル出力時は平文ではなく暗号化バックアップ形式を標準にするか
- 一時ファイル、ログ、クラッシュレポートに金額やメモを残さないようにするか
- スクリーンショットやOSバックアップへの露出をどう考えるか

### 6. 通信時の保護

- MVPではそもそも通信しない構成にするか
- 同期やバックアップを行う場合、エンドツーエンド暗号化を目指すか
- サーバー側で内容を復号できない設計にするか
- 通信先の認証、証明書検証、改ざん検知をどう行うか

### 23. 今すぐ決める最初の論点

最初の1論点は `利用環境の前提` がよい。  
ここが決まると、保存方式、認証方法、同期要否、UI、通知、バックアップ設計の大半が連鎖的に決まるためである。

現時点での候補は次の3つに絞れる。

- スマホ単体で使うローカルファースト家計簿
- PC中心で使うローカル分析寄り家計簿
- 複数端末同期を前提にしたクラウド家計簿

初期MVPとしては、`スマホ単体で使うローカルファースト家計簿` か `PC中心で使うローカル分析寄り家計簿` のどちらかから始めるのが現実的である。  
クラウド同期前提は、セキュリティ、運用、障害対応、コスト、法務の論点が一気に増えるため、後回しのほうが設計を安定させやすい。

### 24. 現時点の合意

- 個人の単一端末利用を前提にする
- スマホ中心で使う
- ローカルファーストを採用する
- 同期はMVPに含めない
- 手入力中心で、毎日入力する運用を支援する
- 主画面候補は `残高推移` を強く意識する
- セキュリティ上の最重要リスクは「データが第三者の手に渡ること」とする
- 保存時暗号化を採用し、鍵はOSの安全領域に保存する方針を採る
- 過去編集は「締めた期間は直接編集不可、修正は取消や訂正取引で表す」を採用する

### 25. 保存方式の整理

`ローカルDB` から `暗号化ファイル` への変換は可能である。  
これは保存方式を完全に切り替えるというより、通常利用の保存先と、バックアップ用の書き出し形式を分けるイメージで考えると分かりやすい。

推奨する構成は次の通り。

- 通常利用時の正本はローカルDBに保存する
- バックアップ時にDB内容をアプリ内部の中間データ構造へ読み出す
- その中間データをJSONなどの決めた形式へ直列化する
- 直列化した内容を暗号化して1つのバックアップファイルとして保存する

バックアップ時の流れは次のようになる。

1. ローカルDBから `accounts`, `transactions`, `entries`, `settings` などを読み出す
2. バックアップ用スキーマへまとめる
3. JSONやMessagePackのような形式へ変換する
4. 鍵導出またはOS保護鍵を使って暗号化する
5. `backup-YYYY-MM-DD.enc` のようなファイルへ保存する
6. 復元時は逆順に、復号 -> 構文検証 -> 整合性検証 -> DB再構築を行う

この方式なら、日常の検索・集計はDBの強みを使え、バックアップや端末移行は暗号化ファイルの扱いやすさを使える。

### 26. バックアップ実装案

考えやすい案は次の3つである。

1. 手動エクスポート型
- ユーザーが明示的に「バックアップを作成」を押したときだけ暗号化ファイルを書き出す
- 実装が簡単で、MVP向き
- 取り忘れのリスクがある

2. 定期自動バックアップ型
- 端末内のアプリ領域やユーザー指定先に、一定間隔で暗号化バックアップを作る
- 取り忘れを防ぎやすい
- ストレージ消費や世代管理の設計が必要

3. 世代管理付きバックアップ型
- 直近N世代を保持し、古い状態へ戻れるようにする
- データ破損や誤編集に強い
- MVPにはやや重いが、中長期では有力

MVPとしては、`手動エクスポート型 + 余裕があれば端末内の定期自動バックアップ` が良い。  
まずは「暗号化バックアップを確実に作れる」ことを優先し、クラウド転送までは考えないほうが安全である。

### 27. 標準搭載する勘定セットの最終案

初期MVPでは、標準勘定を「日常家計を迷わず入力できる最小集合」として固定する。  
細かすぎる分類は避け、まずは残高推移と資金フローの可視化に必要な箱だけを持つ。

#### 27.1 採用する標準勘定

資産

- `asset:cash`
- `asset:bank:main`
- `asset:e_money`
- `asset:saving`

負債

- `liability:card:main`
- `liability:other_payable`

収益

- `income:salary`
- `income:side_job`
- `income:refund`
- `income:point_reward`

費用

- `expense:food`
- `expense:housing`
- `expense:utilities`
- `expense:communication`
- `expense:transport`
- `expense:daily_goods`
- `expense:medical`
- `expense:education`
- `expense:entertainment`
- `expense:social`
- `expense:special`
- `expense:adjustment`

調整用収益

- `income:adjustment`

#### 27.2 各勘定を入れる理由

- `asset:cash`: 現金支払いとATM引き出しを追うために必須
- `asset:bank:main`: 家計の主たるキャッシュポジションを表す中心勘定
- `asset:e_money`: チャージと利用を分離し、支出誤認を防ぐために必要
- `asset:saving`: 貯蓄移動を支出と分けて把握するために必要
- `liability:card:main`: クレジット利用と引落を分離記録する中核
- `liability:other_payable`: 後払い・立替・その他未払の受け皿
- `income:salary`: 主収入の可視化
- `income:side_job`: 給与以外の継続収入の区別
- `income:refund`: 還付や返金系の受け皿
- `income:point_reward`: ポイント還元を独立把握するため
- `expense:*`: 日常生活で迷いにくく、かつ月次レビューに使いやすい粒度
- `income:adjustment` / `expense:adjustment`: 差異調整を通常取引から分離するため

#### 27.3 あえて初期搭載しない勘定

- `asset:brokerage`
- `liability:loan`
- `income:investment`
- `expense:tax`
- `expense:insurance`

これらは必要な人には重要だが、初期標準セットとして入れると、使わないユーザーにとっては迷いの原因になりやすい。  
必要になった時点でユーザーが追加する前提にする。

#### 27.4 命名と階層の方針

- MVPでは階層は最大2段階相当までを基本とする
- 表示名は生活用語にする
- 内部IDは英字固定にして、表示名は後から変更可能にする
- 同一カテゴリ名でも `asset` と `expense` は別勘定として厳密に分離する

表示名の例

- `asset:bank:main` -> `メイン口座`
- `liability:card:main` -> `メインカード未払`
- `expense:daily_goods` -> `日用品`

#### 27.5 ユーザー編集ルール

- デフォルト勘定は自由に表示名変更できる
- デフォルト勘定はアーカイブできる
- デフォルト勘定に紐づく過去取引がなければ削除も許可してよい
- ユーザーは任意の新規勘定を追加できる
- ただし内部 `kind` の変更は、既存取引がある場合は原則禁止する
- 「初期セットへ戻す」で、欠けている標準勘定だけを再追加できるようにする
- 「完全リセット」は別操作にし、既存取引がある場合は強い警告を出す

#### 27.6 この標準勘定セットで実現できること

- 現金、口座、電子マネー、貯蓄の残高推移を見られる
- カード未払を資産とは別に追跡できる
- 日常支出を過不足ない粒度で振り返れる
- 差異調整を通常支出と混同せずに扱える
- 必要な人だけ追加勘定で拡張できる

### 28. 差異調整の設計案

差異調整は、現実残高とアプリ残高のズレを埋めるための専用操作である。  
便利だが多用すると入力漏れを隠すため、通常取引とは明確に分離したほうがよい。

推奨設計は次の通り。

- 通常の取引入力画面とは別に `残高調整` 操作を用意する
- 調整時には「対象勘定」「実残高」「差額」「理由メモ」を必須にする
- 調整取引は通常の支出や収入一覧で目立つラベルを付ける
- 調整履歴は後から必ず追えるようにする
- 調整実行前に「最近の未入力取引がないか確認したか」を促す

内部表現としては、専用の費用・収益勘定を持つと扱いやすい。

- 残高が足りなかった場合: `expense:adjustment`
- 残高が多かった場合: `income:adjustment`

私見では、調整自体は許可するが、頻繁に使う想定ではなく、`2週間に1回の照合時の最終手段` に寄せるのがよい。

### 29. 過去編集ポリシー

採用する方針は、`締めた期間は直接編集不可、修正は取消や訂正取引で表す` である。

この方針の利点

- 残高推移や過去レポートの意味が安定する
- いつ何を直したか追跡しやすい
- 金融データとしての信頼感が高まる

設計案

- ユーザーは2週間に1回程度の照合時に期間を締める
- 締めた期間の取引は直接編集不可にする
- 修正したい場合は `取消` または `訂正取引を追加` のどちらかを選ばせる
- 集計画面では、取消・訂正が起きたことを分かるようにする
- 締め前の期間だけは通常編集を許可する

この方針は、毎日入力のしやすさと、後から見返したときの信頼性の両立に向いている。

### 30. スマホ向け入力フロー

スマホ向けの入力フローでは、`毎日すばやく記録できること` を最優先にする。  
そのため、毎日よく使う入力と、たまにしか使わない管理操作を明確に分ける。

#### 30.1 設計方針

- ホーム画面から1タップで取引入力へ入れる
- よく使う入力は3ステップ以内で完了できるようにする
- 低頻度操作は別導線へ分離し、通常入力を邪魔しない
- 内部プロトコルの複雑さはUIに見せない
- 残高推移を主画面に置きつつ、入力導線は常に目立たせる

#### 30.2 主要画面

1. ホーム
- 主表示は `残高推移`
- 補助表示として `メイン口座残高`, `カード未払`, `今日の入力件数` を置く
- 画面下または右下に `+記録` ボタンを固定配置する

2. クイック入力
- 日常の `支出 / 収入 / 移動 / カード払い` を最短で登録する画面
- 毎日最も多く使う

3. 履歴
- 最近の取引一覧を見る
- 直近未締め期間の取引だけ編集できる

4. 管理
- `カード支払`, `残高照合`, `バックアップ`, `勘定管理`, `設定` を置く
- 低頻度操作をここへ集約する

#### 30.3 最重要導線: クイック入力

クイック入力の最初の分岐は、生活用語ベースで次の4つに絞る。

- `支出`
- `収入`
- `移動`
- `カード払い`

これにより、内部の `expense / income / transfer / credit_expense` へ自然に落とせる。

#### 30.4 支出入力フロー

想定対象

- 現金支払い
- 銀行口座からの即時支払い
- 電子マネー残高からの支払い

入力順

### 33. バックアップ運用の最終仕様

現時点の方針

- MVPでは `手動バックアップ + 端末内自動バックアップ` を採用する
- 手動バックアップは、ユーザーが明示的に安全な持ち出し用ファイルを生成する手段として必須にする
- 自動バックアップは、端末内のアプリ管理領域に限定して保存する
- 自動バックアップも暗号化し、平文保存は行わない
- 自動バックアップはクラウド転送や外部共有を前提にしない
- 自動バックアップは「直近の復旧可能性」を高める補助機能と位置づける

今後さらに決めるべき細部

- 自動バックアップの作成タイミング
- 直近何世代を保持するか
- 保存失敗時の通知方法
- 容量逼迫時の削除ルール
- 管理画面でどこまでバックアップ状態を見せるか

### 35. 残高照合と締め運用の詳細

- 重み: 9/10
- 差異調整や締め済み期間の扱い自体は決まっているが、実際の運用フローはまだ粗い
- 2週間に1回程度の照合をどう促すか、どの勘定を優先して照合するか、照合完了と締めをどう結びつけるかが未決である
- ここが弱いと、プロトコルが正しくても残高の信頼性が落ちやすい
- 実運用では「入力」「照合」「締め」のサイクルが回ることが重要なので、運用設計として明示したい

現時点の方針

- 照合対象の基本セットは `現金 / メイン口座 / 電子マネー / メインカード未払` とする
- `貯蓄口座` や `その他未払` は条件付き照合対象とし、必要に応じてユーザーが追加管理する
- 照合は勘定単位の操作、締めは期間単位の操作として分けて扱う
- 2週間ごとに `未入力確認 -> 残高照合 -> 締め判断` を促す運用を基本とする
- 締めは自動実行せず、ユーザーの明示操作で実行する
- 差異調整は最終手段とし、未入力・日付ズレ・勘定選択ミス・金額ミス・二重入力を確認した後にのみ実行する
- 差異調整は履歴にも集計にも残すが、通常収支とは別枠で表示する

照合対象の理由

- `現金`: 実残高確認が必要で、ズレが起きやすいため
- `メイン口座`: 家計の中心残高であり、最重要の照合対象であるため
- `電子マネー`: チャージと支払いを分けて扱う以上、残高確認価値が高いため
- `メインカード未払`: 将来キャッシュ流出の中心指標であり、照合対象に含めるべきため

今後さらに決めるべき細部

- 照合未完了でも締めを許すか、その場合の警告文言をどうするか
- 差異調整を月次レポートでどう見せるか
- 照合履歴をどこまで一覧表示するか
- 14日ごとの促しを通知で行うか、管理画面表示だけにするか
- 条件付き照合対象の追加・除外をどの設定で管理するか

### 36. 通知とリマインドの扱い

- 通知を採用するなら、頻度、タイミング、文面、オンオフ設定まで一緒に考える必要がある

この方針を採る理由

### 37. 集計キャッシュの要否

- 重み: 8/10

- 毎回フル再計算するのか、補助キャッシュを持つのかで実装の複雑さと応答速度のバランスが変わる
- ここは技術仕様の深部に関わる一方で、ユーザーが感じる快適さにも影響するため、早めに判断したい
- キャッシュを採る場合は、再構築条件、破棄条件、整合性担保の方針も必要になる

現時点の方針

- MVPでは `transactions + entries` を正本とし、集計は原則として再計算する
- 初期MVPでは、月次集計や残高推移のための専用キャッシュを必須実装にしない
- ただし、将来キャッシュを追加しやすい構造は先に意識する
- つまり「まずは正しさ優先、必要になったら表示高速化のためにキャッシュを導入する」方針を採る

この方針を採る理由

- 金融データでは、表示速度よりも数値の正しさを優先したい
- 正本からの再計算は、実装が単純で不整合を起こしにくい
- MVP段階ではデータ量がまだ小さい想定なので、まずは正しい再計算系を完成させる価値が高い
- 一方で、スマホ上で残高推移や月次集計が重くなる可能性はあるため、後からキャッシュを足せる構造は残しておきたい

将来のキャッシュ候補

- 月単位の集計結果キャッシュ
- 勘定ごとの日次残高スナップショット
- ホーム画面用の要約値キャッシュ

### 38. 監査イベントの粒度

- 重み: 8/10
- `audit_events` を持つ方針はあるが、どの操作をどこまで記録するかは未決である
- 締め済みデータの取消や訂正だけを追うのか、設定変更やバックアップ操作まで追うのかで、監査性と実装負荷が変わる
- 実運用では「いつ何が変わったか」が分かることが安心感につながるため、軽視しづらい
- 一方で、ログしすぎると機微情報や保存量の問題も出るので、記録対象を明示的に絞る必要がある

現時点の方針

- 監査イベントはローカルDB内に保持する

- バックアップ時には `audit_events` もバックアップ対象に含める

DB内保持を採る理由

- 正本データがローカルDBであるため、監査イベントも同じ整合性境界で管理しやすい
- 取引更新と監査記録を同時に確定でき、片方だけ失われる事故を減らせる
- スマホアプリ上での検索、履歴表示、復旧、バックアップ復元と相性がよい

### 39. 障害時の復旧導線

- 重み: 8/10
- バックアップ破損、復号失敗、DB破損、復元失敗など、実運用で起こりうる障害にどう向き合うかが未決である
- 技術的には処理失敗で済んでも、ユーザー視点では「戻せるか」「今のデータは消えないか」が最重要になる
- 復旧導線が曖昧だと、セキュアでも怖くて使い続けにくい
- 失敗時の画面メッセージ、ロールバック、既存DB保護、復元前確認の設計が必要である

現時点の方針

- 復旧処理では、既存のローカルDBをすぐ上書きしない
- 復元は必ず `一時DB` に対して行い、最後まで成功した場合にのみ正本へ切り替える
- 復元失敗時は既存DBを保持し、一時DBだけを破棄する
- バックアップ作成失敗時も既存データには一切影響を与えない
- バックアップ破損や復号失敗時は、DBへ一切書き込まず処理を中断する
- ローカルDB破損を検知した場合は、通常起動せず復旧モードへ入る

想定する主な障害種別

- バックアップ作成失敗
- バックアップファイル破損
- バックアップ復号失敗
- 復元途中失敗
- ローカルDB破損

復旧の基本原則

- 既存データを即時上書きしない
- 復旧前に検証できるものは前倒しで検証する
- 失敗時は「現在のデータが守られているか」を明示する
- ユーザーに難しい復旧判断を強制しすぎない

推奨する復元フロー

1. 追加認証
2. バックアップファイル形式検証
3. バージョン確認
4. 復号
5. 本文構造検証
6. プロトコル整合性検証
7. 一時DBへの復元
8. 最終整合性確認
9. 正本DBとの切り替え

障害種別ごとの扱い

- バックアップ作成失敗:
  - バックアップは作成されない
  - 既存データは変更しない
  - 最後に成功したバックアップ日時は維持する

- バックアップファイル破損:
  - 復元前チェックで停止する
  - DBへは一切書き込まない

- バックアップ復号失敗:
  - 復元処理を中断する
  - 既存DBを保持する

- 復元途中失敗:
  - 一時DBのみ破棄する
  - 正本DBは保持する

- ローカルDB破損:
  - 通常起動を止める
  - 復旧モードへ遷移する
  - 復旧モードでは「最新バックアップから復元」「手動バックアップから復元」「終了」を提示する

ユーザー向けに明示したいこと

- 何が失敗したか
- 現在のデータが守られているか
- 次に何をすればよいか

例

- 「バックアップ作成に失敗しました。現在の家計データはそのまま保持されています。」
- 「バックアップを復元できませんでした。現在の家計データは変更されていません。」
- 「保存データに問題が見つかりました。バックアップから復旧できます。」

今後さらに決めるべき細部

- 一時DBの保存場所と削除タイミング
- ローカルDB破損検知をどのタイミングで行うか
- 復旧モードでどこまで詳細エラーを見せるか
- バックアップ作成失敗を通知で知らせるか、管理画面表示だけにするか

### 40. アプリレベル認証のMVP対応

- 重み: 7/10
- 保存時暗号化とOS安全領域の利用は決まっているが、日常利用でアプリ起動時や重要操作時にどこまで再認証を求めるかはまだ未決である
- 覗き見対策として有効だが、入力の手軽さを損なう可能性もある
- 常時ロック、一定時間後ロック、重要操作のみ再認証など、選択肢によって使い勝手が大きく変わる
- 毎日使うアプリなので、セキュリティと摩擦のバランスを丁寧に決める必要がある

現時点の方針

- 通常の記録入力では再認証を必須にしない
- アプリ起動時の常時ロックはMVPでは必須にしない
- 重要操作では再認証を必須にする
- 再認証は `OS生体認証 -> OS PIN/端末認証` の順で利用する
- アプリ独自パスワードはMVPでは持たない

MVPで再認証を必須にする操作

- バックアップ作成
- バックアップ復元
- バックアップ関連設定変更
- 認証・セキュリティ設定変更
- データ初期化や全削除に相当する操作

MVPで再認証を必須にしない操作

- ホーム閲覧
- 履歴閲覧
- 通常の支出入力
- 通常の収入入力
- 通常の移動入力
- カード払い入力
- 未締め期間の軽微な編集

将来強化候補

- 差異調整の確定
- 締め処理の実行
- 締め済み期間の取消
- 訂正取引の追加
- 一定時間非操作後のアプリ再ロック

この方針を採る理由

- 毎日入力の軽さを守れる
- 危険度の高い操作だけを重点的に保護できる
- MVPでの実装複雑性を抑えやすい
- 将来の認証強化余地を残せる

今後さらに決めるべき細部

- 再認証成功後の有効時間を持つか
- 生体認証不可端末でのフォールバック挙動
- 重要操作失敗時のメッセージと再試行導線
- 将来、起動時ロックを設定可能機能として追加するか

### 41. 差異調整の集計上の扱い

- 重み: 7/10
- 専用勘定 `income:adjustment` / `expense:adjustment` を持つ方針は決まっているが、レポートや可視化でどう表示するかは未決である
- 通常の収益・費用と混ぜると実態が見えにくくなる一方、完全に隠すと何が起きたか追いにくい
- 実運用では、差異調整がどれだけ発生しているかは入力品質の指標にもなりうる
- 月次集計や履歴表示で「通常取引とは別枠」として扱うかを明示したい

現時点の方針

- 差異調整は集計に含めるが、通常収支とは明確に分離して表示する
- `income:adjustment` と `expense:adjustment` は、通常の収益・費用カテゴリへ混ぜない
- 差異調整は勘定残高、純資産相当、残高推移には反映する
- ただし、生活費分析や通常カテゴリ別支出分析では別枠として扱う
- 履歴一覧には表示し、通常取引と区別できる専用ラベルを付ける

月次集計での表示方針

- `通常収益合計`
- `通常費用合計`
- `差異調整合計`
- `月末残高`
- `月末未払`

カテゴリ別表示での方針

- 通常カテゴリ一覧には差異調整を含めない
- 差異調整は別セクションで表示する
- 月次レビューでは差異調整の `件数` と `合計額` を出せるようにする

履歴表示での方針

- 差異調整も通常履歴の中に表示する
- ただし `残高調整` などの明確なラベルを付ける
- 将来的には `差異調整のみ` フィルタを持てると望ましい

この方針を採る理由

- 差異調整は現実残高へ合わせるための重要イベントであり、残高計算から外すべきではない
- 一方で、通常の生活支出や収益と混ぜると月次レビューの意味が崩れる
- 別枠表示にすることで、帳簿の信頼性と生活分析の見やすさを両立できる
- 差異調整件数や合計額は、入力漏れや運用品質のシグナルとしても使える

### 42. 勘定編集時の整合ルールの詳細

- 重み: 7/10
- `kind` 変更禁止やアーカイブ方針はあるが、表示名変更、標準勘定復元、ユーザー追加勘定の統合などの細かな挙動はまだ詰めきれていない
- 勘定編集は自由度が高いほど便利だが、過去データの読解性や集計の安定性を壊しやすい
- 実運用では、途中で勘定体系を見直したくなる可能性が高いため、ここを曖昧にしないほうがよい
- 変更履歴を残すか、旧表示名をどこまで保持するかも含めて考える必要がある

現時点の方針

- 表示名変更は許可し、過去取引にも新表示名を反映する
- ただし監査イベントには変更前表示名と変更後表示名の要約を残す
- 既存取引がある勘定の統合はMVPでは提供しない
- アーカイブは許可するが、アーカイブ済み勘定は新規入力候補から外す
- 標準勘定復元は「欠けている標準勘定だけ再追加」に限定する
- 既存取引がある勘定の `kind` 変更は禁止を維持する

この方針を採る理由

- 表示名変更まで禁じると日常運用の自由度が低すぎる
- 一方で統合や種別変更は履歴と集計を壊しやすく、MVPでは重すぎる
- アーカイブ中心にすると、過去データの整合性を守りやすい
- 標準勘定復元を差分追加に限定すると、既存帳簿への破壊的影響を避けやすい

### 43. 自動バックアップの保存先と容量管理

- 重み: 7/10
- 自動バックアップを採用する場合、端末内のどこへ保存するか、何世代まで保持するか、容量逼迫時にどう振る舞うかが未決である
- セキュリティだけでなく、端末ストレージやOS制約とも関わる
- 実装によっては、ユーザーが気づかないうちに保存失敗しているリスクもある
- バックアップ成否の表示や、古い世代の削除ルールまで設計しておきたい

現時点の方針

- 自動バックアップの保存先は `アプリ管理領域` とする
- 手動バックアップの保存先とは分けて扱う
- 自動バックアップは `直近7世代` を保持する
- 容量逼迫時は `最古世代から削除` する
- ただし `最低1世代は必ず残す` ルールを採用する
- バックアップ失敗時は管理画面で明示し、最後に成功した日時を表示する
- 連続失敗時のみ軽い通知を検討し、通常時の通知は増やしすぎない

この方針を採る理由

- アプリ管理領域は自動保存と相性がよく、他アプリからの露出も抑えやすい
- 自動バックアップは持ち出し用ではなく、端末内復旧用の補助機能として整理しやすい
- 7世代あれば、数回前の健全な状態へ戻れる余地がありつつ、保存量が過剰になりにくい
- 最低1世代を守ることで、削除だけ進んで全滅する事故を避けやすい
- 失敗表示は必要だが、通知過多にすると日常利用を邪魔するため、管理画面表示を基本にするのがよい

今後さらに決めるべき細部

- 自動バックアップを作成するタイミング
- 空き容量確認をどの条件で行うか
- バックアップ1世代あたりの想定サイズをどう見積もるか
- 管理画面で世代一覧を見せるか、最新成功日時だけ見せるか

### 44. テスト戦略の具体化

- 重み: 6/10
- テストを重視する方針はあるが、どの層にどのテストを置くかはまだ実装可能な粒度まで落ちていない
- プロトコル整合性、残高再計算、バックアップ往復、締め済み期間制御、暗号化境界など、重要なテスト対象は明確になりつつある
- 実装開始前にテスト戦略を粗くでも固めておくと、設計の抜け漏れを見つけやすい
- 特に金融データでは「壊れ方が静か」だと危険なので、検出重視のテスト設計が必要である

現時点の方針

- MVPでは `ドメインテスト + 永続化境界テスト + バックアップ往復テスト` を中心にする
- 代表的な家計イベント20件はゴールデンテスト化する
- 残高再計算は月次集計と合わせて性質テストを持つ
- 締め済み期間の直接編集禁止は必須テスト対象にする
- 暗号化機能は詳細暗号強度テストより、`平文が意図せず残らないこと` と `往復できること` を優先して検証する

この方針を採る理由

- 金融データでは「正しい計算」と「壊れたときに気づけること」が最重要だから
- MVPでUIまで広くテストするより、帳簿の正しさを守る層へ集中したほうが効果が高い
- ゴールデンテストはプロトコル設計と実装のズレを見つけやすい

### 45. 法務・責任範囲の明文化

- 重み: 6/10
- 個人用MVPとしては優先度がやや下がるが、他者提供や公開を少しでも考えるなら早めに整理しておきたい
- 家計アプリは「安全そうに見える」ため、データ消失時や誤集計時の期待値管理が必要になる
- 金融アドバイスではないこと、保存失敗時の責任範囲、サポート範囲をどう表現するかは後で効いてくる
- 仕様そのものというより、運用・公開フェーズで困らないための設計論点として重要である

現時点の方針

- MVP段階では個人用ツール前提として扱う
- 外部公開時には「金融アドバイスを提供するものではない」ことを明記する
- バックアップは提供するが、データ消失ゼロを保証しない
- サポート範囲や免責は、公開段階で別途文書化する前提にする

この方針を採る理由

- いまはプロダクト仕様の完成が優先であり、対外提供向けの詳細規約はまだ重い
- ただし期待値管理に関わる基本方針だけは先に揃えておいたほうが後で困りにくい

### 46. 長期利用時のデータ成長対策

- 重み: 6/10
- 数年分のデータ蓄積を想定したときの性能維持方針はまだ粗い
- 家計簿は長く使うほど価値が増すため、初期から多少は意識しておきたい
- 古いデータの圧縮、再計算範囲の制御、バックアップサイズ増大への対応など、中長期運用向けの課題がある
- MVPの直近優先度は高くないが、完全に無視すると後で構造変更が大きくなる可能性がある

現時点の方針

- MVPでは古い取引データを削除・圧縮しない
- 長期性能対策は `索引設計 + 必要時の集計キャッシュ追加` で吸収する
- バックアップサイズ増大は、まず世代管理と保存先制御で対応する
- 年単位のアーカイブ機能はMVP対象外とする

この方針を採る理由

- 初期段階から圧縮やアーカイブ機能を入れると、復旧・検証・監査が複雑になる
- 正本を単純に保ったほうが、まずは帳簿の正しさを担保しやすい
- 実際のデータ増加を見てから最適化するほうが合理的である

### 47. 将来拡張の優先順位

- 重み: 5/10
- 将来拡張を後回しにする方針自体は妥当だが、何をどの順序で入れるかの大まかな見取り図はあると判断しやすい
- 銀行連携、OCR、家族共有、投資管理などは性質がかなり異なり、先に入れる機能によって境界設計の重点も変わる
- MVPの直近には影響しないが、「今の設計で何を守るべきか」を判断する補助になる
- 実装前提の優先度表までは不要でも、拡張の方向性は軽く整理しておきたい

現時点の方針

- 拡張優先順位は `通知強化 -> CSV/外部取込 -> 位置起点入力促進 -> 銀行連携/OCR -> 複数ユーザー` を基本線とする
- 複数ユーザーと複数端末同期はMVP後でも後順位に置く
- 投資管理や税務対応は、家計簿の基本運用が安定してから検討する

この方針を採る理由

- まずは単一ユーザーの記録継続性と帳簿信頼性を高める機能から積むほうが価値が大きい
- 同期や複数ユーザーはセキュリティと運用の難度を大きく押し上げるため、後ろに置くのが自然である

### 48. 次にやるべき議論の整理

ここまでの合意を踏まえると、次に進むべき大きな論点は次の順がよい。

1. 監査イベントの粒度
- 締め済み期間の取消、訂正取引、差異調整、バックアップ復元など、どの操作を履歴として残すか決める
- 残高照合と締め運用を固めた今、次は「その運用をどう追跡可能にするか」が自然な流れである

2. 障害時の復旧導線
- バックアップ破損、復号失敗、復元失敗、DB破損時に、既存データをどう守りつつ復旧させるかを決める
- バックアップ方針が固まった今、復旧フローの安全性も詰めやすい

3. アプリレベル認証の範囲
- 日常起動時には求めないのか、重要操作時だけ再認証するのかを明確にする
- 暗号化、鍵管理、バックアップ保護が固まっているため、次は日常アクセス制御の粒度を決める段階である

4. 集計キャッシュの要否
- 正本は再計算可能に保ちつつ、スマホ上の性能をどう担保するかを決める
- 実装寄りの技術論点だが、表示性能に直結するため、早めに判断する価値がある

5. 画面の情報設計
- ホーム画面、入力画面、管理画面の情報設計を再開する
- ただし、UIに先立って運用・監査・復旧の方針をもう少し詰めるほうが手戻りが少ない

### 49. 位置起点通知を導入する場合の追加準備

- 重み: 6/10
- 位置情報SDKを家計簿アプリへ組み込む場合、SDKだけでは完結せず、会計アプリ側で位置取得・滞在判定・POI判定・通知導線を整える必要がある
- ローカルファーストとプライバシー重視の方針を守るには、位置情報をサーバーへ送るのではなく、端末内でPOI判定を完結させる設計が望ましい
- そのためには、ユーザーに生活圏や利用エリアを選んでもらい、必要な地図/POIデータ領域だけを端末に保持する運用が有力である
- これにより、位置情報をサーバー送信せずに、スーパー、コンビニ、ドラッグストア、ATM などの候補判定を端末内で行える可能性が高い

現時点の方針

- 位置起点通知を導入する場合は、`ローカルPOIデータ + 端末内判定` を基本方針とする
- 地図表示より先に、POI判定用データをローカル保持する構成を優先する
- ユーザーに対象領域を選択してもらい、その範囲のPOIデータだけを利用する
- MVPでは対応POIカテゴリを少数に絞る
- 位置起点機能を入れる場合でも、自動記録ではなく「入力促進通知」までに留める

現時点の方針の具体化

- 対象領域はユーザーが1〜3個の生活圏エリアを選ぶ前提にする
- POIカテゴリは `スーパー / コンビニ / ドラッグストア / ATM` の4種を初期対象とする
- 地図表示は必須にせず、まずは判定用POIデータのローカル保持だけで始める
- 生位置履歴は長期保存せず、通知判定に必要な短期情報だけを使う前提にする

この方針を採る理由

- ストレージ使用量とプライバシーのバランスを取りやすい
- 会計アプリの価値は地図表示より入力促進にあるため、POI判定だけ先に成立させれば十分だから
- 生活圏を絞ることで、ローカル判定方式でも現実的なデータ量に抑えやすい

今後さらに決めるべき細部

- 対象領域をどう選ばせるか
- POIデータ形式をどうするか
- POIデータ更新頻度をどうするか
- 複数生活圏をどう扱うか
- ストレージ使用量の上限をどう設計するか
- 位置履歴を端末内でもどこまで保持するか

## Technical Spec Draft

### 1. Goal

初期MVPの技術仕様は、次の5つを満たすことを目的とする。

- 単一端末のスマホ上でローカル完結して動く
- 家計データをローカルDBで扱い、高速に検索・集計できる
- 保存時暗号化により、端末外へデータが漏れた際の可読性を下げる
- バックアップを暗号化ファイルとして安全に持ち出せる
- 締め済みデータを不用意に壊さない

### 2. Runtime Assumptions

- 利用者は個人1名
- 利用端末は単一スマホ
- 同期はMVP対象外
- 通貨は単一通貨
- 入力は手入力中心

### 3. Storage Architecture

保存は次の二層で構成する。

1. 正本ストレージ
- 端末内のローカルDB
- 日常の読取・書込・検索・集計はここで行う

2. バックアップストレージ
- 暗号化バックアップファイル
- エクスポートまたは定期バックアップ時に生成する

この分離により、通常利用時の性能と、バックアップ時の可搬性を両立する。

### 4. Local Database

初期MVPでは、端末内の組み込みリレーショナルDBを前提にする。  
候補実装は SQLite 系で考える。

ローカルDBを採る理由

- 取引件数が増えても検索しやすい
- 集計クエリや期間絞り込みがしやすい
- 締め済み期間や編集可否などの状態管理を持ちやすい
- バックアップ用に必要テーブルをまとめて読み出しやすい

### 5. Data Model

最低限必要な保存対象は次の通り。

#### 5.1 `accounts`

- `account_id`
- `kind`
- `name`
- `parent_account_id`
- `is_default`
- `is_archived`
- `created_at`
- `updated_at`

#### 5.2 `transactions`

- `transaction_id`
- `date`
- `type`
- `status`
- `description`
- `counterparty`
- `external_ref`
- `period_lock_state`
- `created_at`
- `updated_at`

#### 5.3 `entries`

- `entry_id`

### 6. Write Rules

データ書込時は次のルールを守る。

1. すべての取引保存前にプロトコル整合性チェックを行う
2. `transactions` と `entries` は同一DBトランザクションで保存する
3. 締め済み期間に属する取引は直接更新禁止
4. 締め済み期間の修正は `void` または新規訂正取引追加で表す
5. `account.kind` は既存取引がある場合は変更禁止

### 7. Read Model and Recalculation

残高は、原則として `posted` 取引から再計算できる形を正本とする。  
ただし、毎回フルスキャンすると重くなるため、読み取り側では補助的な集計キャッシュを許可する。

方針

- 正本は `transactions + entries`
- 残高・月次集計は再計算可能であることを前提にする
- 表示高速化のためにキャッシュを持ってもよい
- キャッシュは破棄可能で、いつでも再構築できること

### 9. Encryption at Rest

保存時暗号化は採用する。  
目的は、端末外へデータファイルやバックアップファイルが流出した際に、内容を直ちに読めないようにすることである。

方針

- ローカルDB本体はDB層で暗号化する
- バックアップファイルは必ず暗号化する
- 平文JSONや平文CSVを標準バックアップ形式にしない

注意点

- 保存時暗号化だけでは、起動中アプリの覗き見は防げない
- そのため必要に応じてアプリロックや生体認証も併用する

補足方針

- 将来、必要な列に限ってアプリ層で追加暗号化できる余地を残す
- ただし初期MVPでは、集計や検索に必要な列へアプリ層暗号化は入れない

### 10. Key Management

暗号鍵はOSの安全領域で管理する。

方針

- アプリ初回起動時にデータ暗号化用の鍵を生成する
- 生成した鍵素材または鍵をOSの安全領域へ保存する
- アプリは必要時にOS経由で鍵へアクセスする
- 鍵を通常ファイルや平文設定へ保存しない

バックアップ時の考え方

- 端末内DB暗号鍵と、バックアップファイル暗号鍵は分離できる設計が望ましい
- MVPでは、OS安全領域の鍵を使ってバックアップ暗号化鍵を保護するか、バックアップ専用パスフレーズ方式を後から選べる余地を残す

### 10A. Future App-Layer Encryption Boundary

将来、DB層暗号化に加えてアプリ層暗号化を導入したくなっても、変更幅が小さくなるように責務境界を先に定義する。

#### 10A.1 基本方針

- DB層暗号化は「DBファイル全体の保護」を担う
- アプリ層暗号化は「特定フィールドの追加保護」を担う
- この二つは役割が異なるため、併用可能とする

#### 10A.2 アプリ層暗号化の対象候補

将来の追加暗号化対象として現実的なのは次のような自由記述・文脈情報である。

- `description`
- `counterparty`
- `entries.memo`
- ユーザー自由記述メモ

#### 10A.3 アプリ層暗号化の対象にしない列

初期方針として、次の列はアプリ層暗号化しない。

- `amount`
- `date`
- `type`
- `transaction_id`
- `account_id`
- `direction`
- `status`

理由

- 月次集計、残高再計算、検索、絞り込み、整合性検証で広く使うため
- これらを列単位で暗号化すると、家計簿としての使い勝手と保守性が大きく下がるため

#### 10A.4 実装責務の分離

将来の追加暗号化に備え、次の責務分離を推奨する。

1. ドメイン層
- 平文の意味モデルだけを扱う
- 暗号化処理を知らない

2. アプリケーション層
- ユースケースを実行する
- どのフィールドが保護対象かを意識しない

3. 永続化境界
- 保存直前に必要なら対象フィールドを暗号化する
- 読込直後に必要なら対象フィールドを復号する

4. 鍵管理境界
- OS安全領域や鍵取得処理を一箇所に閉じ込める

#### 10A.5 コード上で埋め込みたいルール

文書化だけでなく、コード上で次を強制すると誤りを減らしやすい。

- 暗号化対象フィールドを定数または設定として一元管理する
- 永続化層を通さないDB直接アクセスを禁止または抑制する
- 暗号化対象の保存・読込を共通ユーティリティに限定する
- 対象フィールドの平文永続化を検出するテストを用意する
- `schema_version` に応じて暗号化列の有無を判定できるようにする

#### 10A.6 文書化と実装のどちらが重要か

文書化は必要だが、それだけでは不十分である。  
最も安全なのは、次の三層で守ること。

- 仕様書に責務境界を書く
- コード構造で境界を強制する
- テストで逸脱を検出する

つまり、文書化が出発点で、実際には「実装に埋め込んだルール」と「自動テスト」まで含めて初めて事故を減らせる。

#### 10A.7 MVPでの結論

- 初期MVPではDB層暗号化を正式採用する
- 将来のアプリ層暗号化に備えて、永続化境界と鍵管理境界を分離して実装できるようにする
- アプリ層暗号化は、必要になったときに自由記述系フィールドへ限定して追加する方針を採る

### 10B. Persistence Boundary Interfaces

永続化境界では、`Repository` と `Service` の責務を分ける。

- `Repository`: 保存済みデータへの出し入れを担当する
- `Service`: 複数の保存処理や暗号化処理を束ねたユースケースを担当する

この分離により、DB実装、暗号化、バックアップ、締め制御の責務を混ぜずに保てる。

#### 10B.1 Repository 一覧

1. `AccountRepository`
- 勘定一覧を取得する
- 勘定を追加する
- 勘定を更新する
- 勘定をアーカイブする
- 標準勘定セットを再投入する

2. `TransactionRepository`
- 取引とエントリを保存する
- 取引詳細を取得する
- 条件付きで取引一覧を取得する
- 未締め期間の取引を更新する
- 締め済み取引を `void` にする
- 訂正取引を追加する

3. `ClosureRepository`
- 締め期間を作成する
- 指定日が締め済みか判定する
- 締め履歴を取得する

4. `ReconciliationRepository`
- 残高照合記録を保存する
- 勘定ごとの最新照合状態を取得する
- 差異調整取引に紐づく照合情報を取得する

5. `SettingsRepository`
- アプリ設定を取得する
- アプリ設定を更新する

6. `AuditEventRepository`
- 監査イベントを記録する
- 監査イベント一覧を取得する

#### 10B.2 Service 一覧

1. `BackupService`
- バックアップを生成する
- バックアップを検証する
- バックアップを復元する
- バックアップ作成時の認証、整形、暗号化、保存を束ねる

2. `CryptoService`
- バックアップ本文を暗号化する
- バックアップ本文を復号する
- ラップ鍵方式でデータ鍵をラップ/アンラップする
- 将来必要ならアプリ層暗号化対象フィールドを暗号化する

3. `MonthlySummaryService`
- 月次集計を計算する
- 残高推移やカテゴリ集計を返す
- 必要ならキャッシュ再構築を行う

4. `ClosureService`
- 締め処理を実行する
- 締め前チェックを行う
- 締め後の編集制御ポリシーを適用する

5. `ReconciliationService`
- 実残高と帳簿残高を照合する
- 差額を計算する
- 必要時に差異調整フローへつなぐ

#### 10B.3 設計ルール

- UI層はDBへ直接アクセスしない
- ユースケース層はRepository/Service経由でのみ永続化を行う
- SQLやDB実装詳細はRepository内に閉じ込める
- 暗号化実装はCryptoServiceに閉じ込める
- バックアップ処理はBackupServiceに閉じ込める
- 締め済み期間の編集禁止は、UIだけでなくRepository/Service側でも強制する

#### 10B.4 インターフェース粒度の方針

- 1つの巨大な `DatabaseService` は作らない
- 用途ごとに小さな境界へ分ける
- 将来の差し替えやテストを考え、読む責務と書く責務を必要に応じて分けてもよい

#### 10B.5 この構造を採る理由

- 将来アプリ層暗号化を足すとき、変更点を永続化境界へ閉じ込めやすい
- DB実装差し替えやバックアップ方式変更に強い
- 平文保存事故を減らしやすい
- 単体テストやモック差し替えがしやすい

### 11. Authentication

認証は二層で考える。

1. 端末レベル認証
- OSの端末ロック

2. アプリレベル認証
- 生体認証またはPINによる再入場制御

MVP方針

- 保存時暗号化とOS安全領域の利用を優先する
- アプリレベル認証は任意機能または段階導入でもよい
- ただしバックアップ復元や設定変更などの重要操作では再認証を検討する

### 12. Backup Format

バックアップファイルは、少なくとも次の内容を含む。

- メタデータ
- 勘定マスタ
- 取引
- エントリ
- 締め情報
- 設定のうち復元に必要なもの

メタデータ例

- `backup_version`
- `created_at`
- `app_version`
- `device_id` またはその匿名化識別子
- `schema_version`

### 13. Backup Flow

バックアップ作成フロー

1. DB読取を開始する
2. 一貫したスナップショットを取得する
3. バックアップ用オブジェクトへ整形する
4. 直列化する
5. 暗号化する
6. ファイル保存する
7. 保存後に復号テストまたは整合性検証を行えると望ましい

復元フロー

1. バックアップファイルを選択する
2. 復号する
3. `schema_version` を確認する
4. JSON構造と必須フィールドを検証する
5. プロトコル整合性を検証する
6. 問題なければローカルDBを再構築する

### 13A. Backup Encryption and Protected Operations

バックアップは「第三者の手に渡りやすい持ち出しデータ」とみなし、通常保存より強く慎重に扱う。

#### 13A.1 基本方針

- バックアップファイルは常に暗号化する
- 平文バックアップはMVPでは提供しない
- バックアップ出力時に追加認証を要求する
- バックアップ復元時にも追加認証を要求する
- MVPではバックアップ専用パスフレーズは必須にしない
- 鍵管理はOSの安全領域を前提にする

#### 13A.2 保護したい操作

重要操作として、少なくとも次を保護対象とする。

- バックアップファイルの新規作成
- 既存バックアップファイルの復元
- バックアップ保存先の変更
- バックアップ関連設定の変更

#### 13A.3 追加認証の扱い

追加認証は、生体認証または端末PIN/OS認証ダイアログを利用する。

ルール

- バックアップ出力の直前に再認証する
- バックアップ復元の直前にも再認証する
- 認証失敗時は処理を中断する
- 一定時間内の再試行はOS側の標準制御に従う

MVPでは、アプリ独自パスワードよりもOS標準認証を優先する。  
理由は、実装負荷を抑えつつ、鍵管理をOSの安全領域と自然に結びつけやすいためである。

#### 13A.4 暗号化の責務

バックアップファイル暗号化は、バックアップ生成処理の最後に行う。

流れ

1. DBからバックアップ対象データを読み出す
2. バックアップ用オブジェクトへ整形する
3. 直列化する
4. 直列化済みバイト列全体を暗号化する
5. 暗号化済みファイルとして保存する

この方式では、バックアップファイル単位で一貫して保護できる。

#### 13A.5 鍵の使い方

MVPでは次の考え方を採る。

- バックアップ暗号化のための鍵素材はOS安全領域に依存する
- アプリは平文鍵を永続保存しない
- バックアップ作成時は再認証後に暗号化処理を許可する
- バックアップ復元時は再認証後に復号処理を許可する

実装方式として、バックアップ暗号化は `ラップ鍵方式` を採用する。

#### 13A.6 バックアップファイルの構成

バックアップファイルには次の領域を持たせる。

- 平文ヘッダ
- 暗号化本文
- 整合性検証用メタデータ

平文ヘッダに置いてよいもの

- `backup_version`
- `schema_version`
- `created_at`
- 使用暗号方式識別子

平文で置かないもの

- 家計データ本文
- 勘定名
- 金額
- メモ

#### 13A.7 復元前検証

復元時は次の順で安全確認する。

1. 追加認証
2. バックアップファイル形式検証
3. バージョン確認
4. 復号
5. 本文整合性検証
6. プロトコル整合性検証
7. 既存DB上書き確認
8. 復元実行

#### 13A.8 失敗時のルール

- 認証失敗時はバックアップ作成・復元を中断する
- 復号失敗時は部分復元を行わない
- 構造検証失敗時はDBへ書き込まない
- 復元失敗時も既存DBを破壊しない
- 失敗内容は機微情報を含まない形で表示する

#### 13A.9 UX上の注意

- 「バックアップは暗号化されます」と明示する
- 「復元は現在のデータを上書きする可能性がある」と明示する
- バックアップ出力先の共有や外部送信には注意喚起を入れる
- 復元操作は管理画面の深い導線に置く

#### 13A.10 MVPでの結論

MVPでは、バックアップ保護は次の形で固定する。

- バックアップファイルは常に暗号化する
- バックアップ作成時に追加認証を必須にする
- バックアップ復元時に追加認証を必須にする
- 追加のバックアップ専用パスフレーズは必須にしない
- 鍵管理はOS安全領域を前提にする
- バックアップ暗号化方式は `ラップ鍵方式` を採用する

### 13B. Wrapped-Key Backup Design

バックアップ暗号化は、本文を直接OS鍵で暗号化するのではなく、本文用のデータ鍵を別に生成し、そのデータ鍵をOS安全領域に紐づく保護鍵でラップする。

#### 13B.1 鍵の役割

1. データ鍵
- 各バックアップファイルごとに新しく生成するランダム鍵
- バックアップ本文の暗号化にのみ使う

2. ラップ鍵
- データ鍵を保護するための鍵
- OS安全領域に保護された鍵素材に依存する

この分離により、本文暗号化と鍵保護を切り分けられる。

#### 13B.2 作成時フロー

バックアップ作成時は次の順で処理する。

1. ユーザーに追加認証を要求する
2. DBから一貫したスナップショットを取得する
3. バックアップ本文オブジェクトを生成する
4. バックアップごとのランダムなデータ鍵を生成する
5. データ鍵で本文を暗号化する
6. ラップ鍵でデータ鍵をラップする
7. 平文ヘッダ、ラップ済み鍵、暗号化本文、整合性情報を1ファイルへ保存する

#### 13B.3 復元時フロー

復元時は次の順で処理する。

1. ユーザーに追加認証を要求する
2. バックアップファイルのヘッダを読む
3. ラップ済みデータ鍵を取得する
4. ラップ鍵でデータ鍵をアンラップする
5. データ鍵で本文を復号する
6. バージョン・構造・整合性を検証する
7. 問題なければDBを再構築する

#### 13B.4 バックアップファイル構造

バックアップファイルは概念上、次の構造を持つ。

```json
{
  "header": {
    "backup_version": "1",
    "schema_version": "1",
    "created_at": "2026-03-16T10:00:00Z",
    "encryption_scheme": "wrapped-key-v1"
  },
  "wrapped_key": {
    "wrap_scheme": "os-keystore-v1",
    "wrapped_data_key": "<binary-or-base64>"
  },
  "payload": {
    "ciphertext": "<binary-or-base64>",
    "nonce": "<binary-or-base64>"
  },
  "integrity": {
    "tag": "<binary-or-base64>"
  }
}
```

実際のファイル形式はJSON固定でなくてもよいが、論理構造はこの分割を守る。

#### 13B.5 この方式を採る理由

- バックアップ本文を毎回新しい鍵で暗号化できる
- OS安全領域の保護を利用しつつ、将来の鍵方式変更に備えやすい
- 将来的にバックアップ専用パスフレーズを追加しやすい
- 端末依存鍵方式より、設計の拡張余地が大きい

#### 13B.6 MVPでの割り切り

MVPでは次の割り切りを認める。

- ラップ鍵は単一端末前提でよい
- 別端末移行用の完全な鍵移送はまだ提供しない
- バックアップ専用パスフレーズは将来追加可能な拡張点に留める

つまり、MVPのラップ鍵方式は「将来拡張しやすい構造を先に採りつつ、運用は単一端末前提で始める」方針である。

#### 13B.7 将来拡張しやすい点

将来、次の拡張を入れやすい。

- バックアップ専用パスフレーズ導入
- 端末移行用の鍵エクスポート手順
- 複数バックアップ形式への対応
- 暗号方式のバージョン更新

#### 13B.8 注意点

- OS安全領域に依存する以上、端末侵害時の完全防御はできない
- 実行中アプリ権限を奪われた場合の防御には限界がある
- バックアップの可搬性は、将来の鍵移行設計なしには限定的である

### 14. Closed Period Handling

締め済み期間の扱いは、アプリ信頼性の中心仕様として固定する。

ルール

- 取引日が締め済み期間に属する場合、直接編集不可
- 取消は `status = void` または専用取消操作で表現する
- 訂正は新規取引追加で表現する
- 締め解除はMVPでは原則提供しない、または強い警告付きの管理者操作に限定する

### 15. Balance Reconciliation

残高照合は低頻度の運用機能として持つ。

ルール

- 勘定ごとに照合記録を残す
- 推奨頻度は2週間に1回
- 差額がある場合のみ差異調整を許す
- 差異調整時は理由メモ必須
- 差異調整取引は集計上も識別可能にする

### 16. Logging and Privacy

ログには機微情報を極力出さない。

禁止したいもの

- 金額の平文出力
- メモ欄の全文出力
- バックアップ内容のデバッグ表示
- 鍵や復号素材のログ出力

許容しうるもの

- エラーコード
- 取引件数
- スキーマバージョン
- 匿名化済みイベント種別

### 17. Migration Strategy

将来スキーマ変更が起きる前提で、マイグレーションを考慮する。

方針

- `schema_version` をDBとバックアップに持つ
- アプリ起動時に必要なDBマイグレーションを実行する
- バックアップ復元時もバージョン変換の入口を用意する

### 20. Threat Model

このアプリのセキュリティ設計では、「どの脅威を主に防ぎ、どの脅威は限定的にしか防げないか」を先に明示する。

#### 20.1 守りたいもの

- 家計取引データ
- 勘定残高
- メモや説明文に含まれる生活情報
- 暗号化バックアップファイル
- 締め済み期間を含む履歴の完全性

#### 20.2 主要な脅威

1. 端末外へデータファイルが流出する
- 端末紛失
- バックアップファイルの誤共有
- ファイルコピーによる持ち出し

2. 第三者が端末を直接操作する
- 家族や他人による覗き見
- 端末ロック解除後の無断閲覧

3. アプリ権限で悪意あるコードが動く
- アプリ脆弱性の悪用
- 依存ライブラリやSDK経由の侵入
- マルウェア感染端末
- root化・脱獄端末での保護回避

4. データが意図せず漏れる
- デバッグログ
- クラッシュレポート
- 平文エクスポート
- 一時ファイル

5. データが不正または誤操作で壊れる
- 締め済み期間の直接編集
- 誤削除
- 不正バックアップ復元

#### 20.3 MVPで主に防ぐ対象

MVPで主に防ぐのは次の脅威とする。

- 端末外へ流出したファイルの平文閲覧
- 日常利用時の軽度な覗き見
- 誤操作や無自覚編集による履歴破壊
- ログやバックアップ経由の意図しない漏えい

#### 20.4 MVPで限定的にしか防げない対象

次の脅威には一定の軽減策は取るが、完全防御は難しい。

- 端末自体が侵害されている場合
- アプリ実行中にアプリ権限で悪意あるコードが動く場合
- root化・脱獄環境での深い解析

理由

- アプリが利用可能な復号済みデータは、同権限の悪意あるコードにも到達されうるため
- 保存時暗号化は「保存データの流出」には強いが、「実行中メモリの保護」には限界があるため

#### 20.5 想定する具体的な発生場面

- 紛失した端末からアプリデータ領域が抜き出される
- ユーザーがバックアップファイルを誤って第三者へ送る
- 開発用ログに金額やメモを出してしまう
- 外部入力処理の不備で異常データが混入する
- ライブラリ経由で想定外のコードが実行される
- root化・脱獄端末でローカル保護が弱まる

#### 20.6 採る対策

1. 保存データ保護
- ローカルDBを暗号化する
- バックアップファイルを必ず暗号化する
- 鍵はOSの安全領域で管理する

2. アクセス制御
- 重要操作では再認証を検討する
- 生体認証またはPINによる再入場制御を段階導入可能にする

3. 実装面の攻撃面縮小
- 不要なSDKや通信機能を極力入れない
- 外部入力はすべて検証する
- ログに機微情報を残さない

4. 完全性保護
- 締め済み期間は直接編集不可にする
- 取消と訂正取引で修正する
- 監査イベントを保持する

5. 運用上の軽減策
- root化・脱獄環境では警告を出す余地を残す
- バックアップ出力と復元は慎重な導線に置く

#### 20.7 設計上の結論

このアプリは、`保存データの保護` と `履歴完全性の維持` を強く守る設計にする。  
一方で、`アプリ実行中に同権限の悪意あるコードが動く状況` は完全には防げない前提を置き、攻撃面の縮小と被害の局所化を目指す。

## Assumptions

- ターゲットは、家計簿をつけたいが、単純なカテゴリ集計だけでは不十分だと感じる個人ユーザーである。
- 初期段階では法人会計レベルの厳密性ではなく、家計判断に役立つ説明可能性を優先する。
- ユーザーは会計学の専門知識を必ずしも持たないため、UIには生活用語とテンプレート入力が必要である。
- 複式簿記をそのまま手入力させると負荷が高いため、内部モデルと入力体験は分離する必要がある。
- クレジットカード、口座振替、貯蓄移動のような「時間差のあるお金の流れ」を正しく扱えることが、このアプリの差別化要因になる。
- 最初は単一ユーザー・単一通貨を前提にしたほうが検証しやすい。
- MVPでは自動連携なしでも、手入力または簡易インポートで十分に価値検証できる可能性がある。

## Validation Plan

1. 家計イベントを勘定テンプレートへ落とせるか検証する
- 給与受取、現金払い、カード払い、口座間振替、積立、立替精算など、日常の代表イベントを20件程度列挙する。
- 成功判定: 主要イベントの大半が、無理のない勘定テンプレートで表現できる。

2. ユーザーに見せる概念モデルを簡略化できるか確認する
- 「資産・負債・収益・費用」という内部構造を、UIでは何語彙に置き換えると理解しやすいかを整理する。
- 成功判定: 会計知識が薄い人にも、入力時に迷いが少ないラベル案を作れる。

3. 可視化アウトプットの価値を試す
- 同じ1か月分データについて、通常のカテゴリ集計画面と、勘定ベースの資金フロー表示を並べて比較する。
- 成功判定: 「お金がどう流れたか」「未払いがどれだけ残っているか」が後者のほうが速く説明できる。

4. クレジットカード処理の整合性を確かめる
- 利用時と引き落とし時を分けて扱っても、残高・負債・費用認識が破綻しないかを手計算で確認する。
- 成功判定: 月またぎのカード利用を含んでも、資産残高と未払残高を一貫して追跡できる。

5. MVPに必要な最小画面を定める
- 取引入力、勘定一覧、残高推移、資金フロー表示の4画面で価値が成立するかを検討する。
- 成功判定: 余計な機能を足さずに、「記録する」「流れを見る」「次月に備える」が一通り回る。
