# CHOBO 仕様書 v0.1

## 1. 目的

CHOBO は、家計簿を単なる支出集計ではなく、会計学の勘定構造を使って「お金がどこから来て、どこへ流れ、何が将来のキャッシュを圧迫するか」を可視化する個人向け家計管理アプリである。

このアプリの主目的は、次の問いに答えやすくすることにある。

- 今月のお金は何に使われたか
- どの資産にお金が滞留しているか
- クレジットカード利用分の未払がどれだけ残っているか
- 貯蓄や投資への振替がどれだけ実行できているか

本仕様書は、MVP の中核となる内部プロトコル、保存方針、入力運用、保護方針を定義する。

着手順と issue の対応は [`chobo-roadmap.md`](chobo-roadmap.md) にまとめてある。

## 2. 製品の位置づけ

CHOBO は、会計の厳密な表現をそのままユーザーに押し付けるのではなく、生活用語で入力できる UI と、会計的に一貫した内部表現を分離して持つ。

ユーザー向けの表現は生活用語に寄せる。

- 収入
- 支出
- 移動
- カード払い
- カード支払
- 残高照合

内部では、勘定科目と増減方向で整合性を持つプロトコルに変換する。

## 3. MVP の非目的

MVP では次を扱わない。

- 複数通貨
- 銀行 API やカード明細 API との自動同期
- 家族共有の個人別持分管理
- 厳密な会計期間締め処理
- 減価償却や税計算
- レシート OCR
- 位置起点の自動記録
- クラウド前提の多端末同期

## 4. 設計原則

CHOBO の設計原則は次の通りである。

- 単一端末の個人利用を前提にする
- スマホ中心で毎日入力しやすいことを優先する
- 初期実装は Android 先行とし、iOS へ展開しやすい共通コードベースを前提にする
- Tauri の採用は現時点では見送る
- ローカルファーストを採用する
- 保存データは端末外流出を前提に保護する
- 内部表現は会計的に一貫させるが、UI では会計用語を露出しすぎない
- 通常入力は軽く、保護が必要な操作だけを重くする
- 取消と訂正で履歴を守り、締めた期間は直接壊さない
- 正しさを優先し、表示高速化は後から足せるようにする

## 5. ドメインモデル

### 5.1 Account

`Account` は、残高または累計を持つ最小単位の勘定である。

属性は次の通り。

- `account_id`: 一意識別子
- `name`: 表示名
- `kind`: `asset | liability | income | expense | equity`
- `parent_account_id`: 任意。階層化用
- `currency`: 初期 MVP では単一通貨
- `is_archived`: 利用停止フラグ

`equity` は内部表現として予約する。MVP では、期間損益の厳密な締め処理は後回しにし、可視化上は `asset - liability` を純資産相当として扱ってよい。

### 5.2 Entry

`Entry` は 1 つの勘定に対する増減を表す最小記録単位である。

属性は次の通り。

- `account_id`
- `direction`: `increase | decrease`
- `amount`: 正の金額
- `memo`: 任意

### 5.3 Transaction

`Transaction` は複数の `Entry` を束ねた家計上の 1 イベントである。

属性は次の通り。

- `transaction_id`
- `date`
- `type`
- `status`: `posted | pending | void`
- `entries`
- `counterparty`: 任意
- `description`: 任意
- `external_ref`: 任意

1 つの `Transaction` は、必ず 2 件以上の `Entry` を持つ。

## 6. 勘定種別と残高意味

勘定種別ごとの増減の意味を固定する。

| kind | `increase` の意味 | `decrease` の意味 |
| --- | --- | --- |
| `asset` | 資産が増える | 資産が減る |
| `liability` | 負債が増える | 負債が減る |
| `income` | 収益が増える | 収益を打ち消す |
| `expense` | 費用が増える | 費用を打ち消す |
| `equity` | 純資産が増える | 純資産が減る |

この定義により、借方・貸方を UI に露出しなくても、勘定種別と増減方向だけで処理できる。

## 7. 標準取引型

MVP の `type` は次の 5 種類に固定する。

1. `income`
- 外部から家計へ資金が入る取引
- 例: 給与、還付、副収入、ポイント還元

2. `expense`
- 家計資産を使って費用が発生する取引
- 例: 現金払い、デビット払い、即時引落

3. `transfer`
- 家計内の資産勘定同士で資金を移す取引
- 例: 普通預金から現金、普通預金から積立口座

4. `credit_expense`
- 費用は確定するが、現時点では現金が出ず、負債が増える取引
- 例: クレジットカード利用、後払い決済

5. `liability_payment`
- 既存負債を資産から返済する取引
- 例: カード引落、ローン返済

## 8. 正規の取引形

MVP では、各 `type` は 1 つの正規形に固定する。
保存される `Transaction` は、いずれもこの正規形に正規化できる必要がある。

### 8.1 Income

- `income` 勘定: `increase`
- `asset` 勘定: `increase`

例:

- `income:salary` `increase` 300000
- `asset:bank:main` `increase` 300000

### 8.2 Expense

- `asset` 勘定: `decrease`
- `expense` 勘定: `increase`

例:

- `asset:cash` `decrease` 2500
- `expense:food` `increase` 2500

### 8.3 Transfer

- 送金元 `asset`: `decrease`
- 送金先 `asset`: `increase`

例:

- `asset:bank:main` `decrease` 50000
- `asset:saving` `increase` 50000

`transfer` は純支出ではないため、`expense` を増やしてはならない。

### 8.4 Credit Expense

- `liability` 勘定: `increase`
- `expense` 勘定: `increase`

例:

- `liability:card:main` `increase` 4200
- `expense:food` `increase` 4200

この時点では `asset` を減らさない。キャッシュ流出は後続の `liability_payment` で表現する。

### 8.5 Liability Payment

- `asset` 勘定: `decrease`
- `liability` 勘定: `decrease`

例:

- `asset:bank:main` `decrease` 4200
- `liability:card:main` `decrease` 4200

## 9. 整合性ルール

投入される `Transaction` は次の条件を満たす。

1. 金額はすべて正数である
2. `entries` は 2 件以上である
3. `type` ごとの標準形に従い、勘定種別ごとの増減が内部的に釣り合っていること
4. `posted` は残高と月次集計の計算対象である
5. `pending` は未計上だが、保存するなら正規形を満たしている必要がある
6. `void` は取消として扱い、残高と月次集計の計算対象外である
7. `transfer` は同一通貨の `asset` 間に限る
8. `credit_expense` は必ず `liability` を含む
9. `liability_payment` は必ず既存負債を減らす

厳密な借方・貸方は UI に露出しないが、保存前バリデーションでは標準形の組み合わせが会計的に矛盾していないことを必須とする。
`void` 化した取引は、監査のために履歴として保持するが、残高計算では適用しない。

`Transaction` は保存前にバリデーションされ、整合性を満たさないものは保存しない。

## 10. 残高と集計

残高は、時系列順に `posted` のみを適用して算出する。`pending` は未計上、`void` は取消として扱う。

- `asset` 残高 = `increase` 合計 - `decrease` 合計
- `liability` 残高 = `increase` 合計 - `decrease` 合計
- `income` 累計 = `increase` 合計 - `decrease` 合計
- `expense` 累計 = `increase` 合計 - `decrease` 合計

月次可視化では、少なくとも次を出す。

- `net_assets = total_assets - total_liabilities`
- `cash_out_expenses`
- `accrued_expenses`
- `liability_due`

`cash_out_expenses` は、現金または預金減少を伴った費用の合計である。  
`accrued_expenses` は `credit_expense` により発生した未払費用の合計である。  
`liability_due` は未返済負債残高である。

## 11. UI と内部型の対応

| UI 入力種別 | 内部 `type` | 必須選択項目 |
| --- | --- | --- |
| 収入 | `income` | 入金先資産勘定、収益カテゴリ |
| 支出 | `expense` | 支払元資産勘定、費用カテゴリ |
| 移動 | `transfer` | 送金元資産勘定、送金先資産勘定 |
| カード払い | `credit_expense` | 負債勘定、費用カテゴリ |
| カード支払 | `liability_payment` | 支払元資産勘定、対象負債勘定 |

UI は生活用語を使い、内部で勘定ベースのプロトコルへ変換する。

## 12. 標準勘定セット

MVP では、日常家計で迷わず入力できる最小集合を標準搭載する。
これらは `is_default = true` の標準勘定として扱う。

### 12.1 資産

- `asset:cash`
- `asset:bank:main`
- `asset:e_money`
- `asset:saving`

### 12.2 負債

- `liability:card:main`
- `liability:other_payable`

### 12.3 収益

- `income:salary`
- `income:side_job`
- `income:refund`
- `income:point_reward`
- `income:adjustment`

### 12.4 費用

- `expense:food`
- `expense:housing`
- `expense:utilities`
- `expense:communication`
- `expense:transport`
- `expense:daily_goods`
- `expense:medical`
- `expense:education`
- `expense:entertainment`
- `expense:social`
- `expense:special`
- `expense:adjustment`

### 12.5 勘定編集ルール

- 表示名変更は許可する
- アーカイブは許可する
- `kind` 変更は、既存取引がある場合は禁止する
- ユーザーは任意の新規勘定を追加できる
- 既存取引がある勘定の統合は MVP では提供しない
- 標準勘定の `account_id` と `kind` は固定し、ユーザー編集の対象にしない
- 標準勘定復元は「欠けている標準勘定だけ再追加」に限定する
- 標準勘定復元は、既存のユーザー勘定やカスタム勘定を変更しない
- すでに存在する標準勘定は、復元時に上書きしない

## 13. 入力運用

入力はスマホ中心で、毎日すばやく記録できることを優先する。

### 13.1 クイック入力

クイック入力の最初の分岐は次の 4 つに絞る。

- 支出
- 収入
- 移動
- カード払い

クイック入力では、1 回の入力で 1 つの正規形だけを作る。
カード支払は日常入力の流れから外し、照合や管理画面側の操作として扱う。

### 13.2 カード支払

カード支払は日常入力ではなく管理操作として別導線に置く。

### 13.3 テンプレート

- よく使う取引はテンプレート保存できる
- 前回と同じカテゴリや支払元を候補として出す
- 定型入力は再利用しやすくする

### 13.4 編集方針

- 未締め期間の取引は直接編集できる
- 締め済み期間の取引は直接編集不可とする
- 締め済み期間は `void` か訂正取引で修正する

## 14. 残高照合と締め

残高照合は勘定単位、締めは期間単位で扱う。

### 14.1 照合対象

基本対象は次の 4 つである。

- `asset:cash`
- `asset:bank:main`
- `asset:e_money`
- `liability:card:main`

### 14.2 照合の流れ

1. 対象勘定を選ぶ
2. アプリ上残高を表示する
3. 実際残高を入力する
4. 差額を表示する
5. 必要なら差異調整へ進む
6. 照合完了として記録する

照合が未完了でも締め操作自体は可能とするが、未完了の対象勘定があれば警告を出す。
締めは、照合を完了させるための強制操作ではなく、ユーザーが明示的に行う期間固定操作である。

### 14.3 締め方針

- 2 週間に 1 回程度の照合サイクルを想定する
- 締めはユーザーの明示操作で行う
- 締めた期間は直接編集できない
- 修正は取消または訂正取引で表す
- 締め後に差異が見つかった場合は、締め後の新規取引または訂正取引で追跡する

## 15. 差異調整

差異調整は、現実残高とアプリ残高のズレを埋める専用操作である。

推奨設計は次の通り。

- 残高調整は通常取引入力と分ける
- 対象勘定、実残高、差額、理由メモを必須にする
- 通常の費用・収益とは別ラベルで表示する
- 調整履歴は後から必ず追えるようにする

内部表現としては、次の勘定を使う。

- 残高が足りなかった場合: `expense:adjustment`
- 残高が多かった場合: `income:adjustment`

差異調整は許可するが、頻繁に使う前提ではない。
差異調整を保存したら、監査イベントを必ず記録する。

## 16. 監査と訂正

重要な状態変化は監査イベントとして残す。

### 16.1 記録対象

- 取引作成
- 未締め期間の取引更新
- 残高照合の完了
- 締め済み期間の取引取消
- 訂正取引の追加
- 差異調整の実行
- 期間の締め
- 勘定作成
- 勘定更新
- 勘定アーカイブ
- 標準勘定セット復元
- バックアップ作成
- バックアップ復元
- 重要設定変更

### 16.2 記録しないもの

- 画面表示
- 一覧フィルタ変更
- 入力途中の一時状態
- 下書きや未保存状態

### 16.3 記録の粒度

監査イベントには、変更前後の全文を持たせない。  
必要最小限の要約、対象 ID、日付、金額、状態変化のみを持たせる。

## 19. アプリ内の安全設計

次の方針を採る。

- 不正な `Transaction` が残高を壊さないよう、保存前バリデーションを必須にする
- 不要な SDK や通信機能は増やしすぎない
- ログに機微情報を残さない
- インポート経由のデータは厳密に検証する
- 誤削除防止のため、削除より `void` を優先する

## 20. 通知

MVP では通知は少数に絞る。

採用候補は次の 3 系統である。

- 入力忘れの軽い通知
- 2 週間ごとの照合リマインド
- バックアップ連続失敗通知

通知文面には、金額、店舗名、位置名などの機微情報を含めない。

## 21. 非機能の優先順位

MVP の優先順位は次の通りである。

1. 残高と履歴の正しさ
2. 締め済みデータの保全
3. 暗号化バックアップ
4. 毎日入力のしやすさ
5. 集計表示の応答速度

速度最適化は重要だが、正しさを壊してまで優先しない。

## 26. 設計改善の論点

本章は、MVP では即時に結論を固定しないが、会計・家計簿アプリとして今後詰めるべき設計論点を整理する。

### 26.1 MVP で今すぐ直す項目

この節は、実装着手前に仕様として固定しておくべき論点をまとめる。

#### 26.1.1 複式整合性と残高検証

取引の保存可否を決める最重要ルールであり、残高計算の前提になる。

- `Transaction` の各 `Entry` は金額整合が取れている必要がある
- 取引型ごとの標準形だけでなく、保存前バリデーションの条件を明示する
- 残高計算の前提となる整合性ルールを数式レベルで定義する

#### 26.1.2 カード利用と未払管理

利用時点の費用認識と、後日の引落処理を分けて扱うための論点である。

- `credit_expense` と `liability_payment` の役割を分けて定義する
- カード利用時点の費用認識と引落時点の資金移動を混同しない
- 未払残高、締日、引落日、支払予定額の表示を検討する

#### 26.1.3 照合と締めの運用

照合と締めの順序が曖昧だと、修正不能なデータを作りやすくなる。

- 残高照合と期間締めの前後関係を明確にする
- 照合未完了時に締められるかを定義する
- 締め後に差異が見つかった場合の扱いを決める

#### 26.1.4 差異調整の勘定設計

現実残高との差分をどの勘定で吸収するかを、資産と負債で分けて定める必要がある。

- 資産勘定と負債勘定で差異調整の符号が逆になる点を明示する
- 調整専用勘定をどの勘定種別に持つかを定義する
- 調整が通常の費用・収益に紛れない表示を用意する

#### 26.1.5 訂正・取消・`void`

締め済みデータの修正方法を統一しないと、監査性と復元性が損なわれる。

- `void` と訂正取引の使い分けを定義する
- 締め済み期間の修正手段を操作単位まで詰める
- 監査イベントと履歴表示の関係を決める

#### 26.1.6 勘定編集と標準勘定管理

勘定の増減や復元を安全に扱うため、編集権限と標準セットの境界を明確にする。

- 表示名変更、アーカイブ、削除、統合、`kind` 変更の扱いを分ける
- 標準勘定セットが「初期投入用」か「常時保証される必須セット」かを明示する
- 標準勘定の復元とバージョン移行の責務を分離する
