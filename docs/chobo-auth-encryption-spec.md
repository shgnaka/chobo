# CHOBO 認証・暗号化仕様

## 1. 目的

この文書は、CHOBO における保存時暗号化、鍵管理、追加認証、機微情報の扱いを定義する。

対象は次のとおり。

- ローカル DB の保存時暗号化
- 鍵の保管と取得
- 重要操作時の再認証
- アプリ内の機微情報露出抑制

## 2. 基本方針

- 正本データはローカル DB に置く
- DB は保存時暗号化を行う
- 鍵は OS の安全領域で管理する
- 通常入力では再認証を必須にしない
- 重要操作のみ追加認証を要求する
- 平文の鍵や復号素材を永続保存しない
- 保存時暗号化だけで、実行中アプリの覗き見は完全には防げない

## 3. 保存時暗号化

### 3.1 対象

保存時暗号化の対象はローカル DB 全体とする。

### 3.2 目的

- 端末外へ DB ファイルが流出した場合の可読性を下げる
- 平文での直接閲覧リスクを下げる
- バックアップやファイルコピー時の漏えい耐性を上げる

### 3.3 方針

- DB ファイル全体を保護する
- 集計や検索に必要な列を列単位で暗号化しすぎない
- 将来、自由記述系フィールドだけを追加暗号化できる余地を残す

## 4. 鍵管理

### 4.1 基本

- 初回起動時にデータ暗号化用の鍵を生成する
- 鍵素材または鍵を OS の安全領域へ保存する
- アプリは必要時に OS 経由で鍵へアクセスする
- 鍵を通常ファイルや平文設定へ保存しない

### 4.2 分離

- 端末内 DB の鍵と、バックアップ用の鍵は分離できる設計にする
- 将来の鍵方式変更に備えて、鍵管理境界を独立させる

### 4.3 バックアップ用の暗号前提

バックアップ仕様では、次の前提を採る。

- 本文暗号化方式は `aes-gcm-v1`
- ラップ方式は `os-secure-storage-v1`
- 端末内で保持する少量のマスター鍵素材は `flutter_secure_storage` に預ける
- `aes-gcm-v1` の暗号化は、本文と整合性タグをまとめて扱う
- 主要な暗号境界は永続化境界に閉じ込める

`aes-gcm-v1` は、CHOBO のバックアップ本文や DB の保存時暗号化に使う共通前提として扱う。  
ラップ鍵と本文鍵は用途を分け、同じ鍵素材を流用しない。

## 5. アプリ認証

### 5.1 方針

- 通常の記録入力では再認証を必須にしない
- 重要操作では再認証を必須にする
- 再認証は OS の生体認証、または OS PIN/端末認証を使う
- アプリ独自パスワードは MVP では必須にしない

### 5.2 重要操作の例

- バックアップ作成
- バックアップ復元
- バックアップ関連設定変更
- 認証・セキュリティ設定変更
- データ初期化や全削除

### 5.3 将来候補

- 一定時間非操作後のアプリ再ロック
- 起動時ロックの標準化
- 編集権限と閲覧権限の分離

## 6. 機微情報の扱い

### 6.1 ログ

- ログに金額やメモ全文を残さない
- 復号素材や鍵素材をログ出力しない
- デバッグ用出力は最小限にする

### 6.2 通知

- 通知文面に金額、店舗名、位置名を含めない
- ロック画面の露出を前提に、抽象的な文面にする

### 6.3 画面表示

- 画面外への露出を減らす
- 自由記述の扱いは必要最小限にする

## 7. 実装境界

実装では次の責務を分ける。

- ドメイン層: 平文の意味モデルのみを扱う
- アプリケーション層: ユースケースを実行する
- 永続化境界: 保存直前・読込直後の暗号化と復号を扱う
- 鍵管理境界: OS 安全領域とのやり取りに閉じ込める

## 8. 保護方針の限界

次の脅威は完全防御が難しい。

- 端末自体が侵害されている場合
- アプリ実行中に悪意あるコードが同権限で動く場合
- root 化や脱獄環境で深い解析が行われる場合

このため、CHOBO は「保存データの保護」と「露出面の縮小」を主眼に置く。
