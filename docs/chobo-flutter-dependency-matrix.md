# CHOBO Flutter 依存候補マトリクス

## 1. 目的

この文書は、Flutter で CHOBO を作るときに入りやすい依存候補を、採用ポリシーに沿って初期分類したメモである。

ここでの分類は固定ではなく、実装前に再評価する前提で使う。

## 2. 前提

- 対象は Android 先行、iOS 追随のスマホアプリ
- UI は Flutter の単一コードベースで持つ
- 保存はローカル DB 中心
- 保存時暗号化と追加認証を扱う
- Tauri は採用しない

## 3. 必須候補

### 3.1 Flutter / Dart

- 分類: `must`
- 理由: CHOBO の共通 UI 基盤そのもの
- 自前実装の余地: なし

### 3.2 ローカル DB アクセス層

- 分類: `must`
- 代表候補: `sqflite`, `drift`, `sqlite3` 系
- 理由: 正本データを端末内で保存し、検索・集計・締め制御を行うため
- 検討ポイント:
  - SQL をどこまで自前で書くか
  - 型安全性を重視するか
  - Android 先行での実装速度を重視するか

### 3.3 保存時暗号化 / 鍵保管の境界

- 分類: `must`
- 代表候補: secure storage 系プラグイン、暗号化ライブラリ
- 理由: 端末外流出への耐性が必要で、平文保存は避けるため
- 検討ポイント:
  - OS 安全領域を直接使うか
  - アプリ層で暗号化するか
  - 鍵をどこまでプラグインに閉じ込めるか

### 3.4 バックアップ入出力

- 分類: `must`
- 代表候補: `path_provider`, file picker 系
- 理由: 暗号化バックアップの作成と復元に必要
- 検討ポイント:
  - Android のみなら SAF 中心で足りるか
  - iOS 展開時の共有導線をどう揃えるか

## 4. あると楽な候補

### 4.1 ルーティング

- 分類: `should`
- 代表候補: `go_router`
- 理由: 画面数が増える家計簿 UI で、画面遷移を整理しやすい
- いまの判断: 採用
- 自前実装との比較:
  - 小規模なら標準 Navigator でも足りる
  - ただし今後の画面追加を考えると、路線は決めやすい

### 4.2 状態管理

- 分類: `should`
- 代表候補: `flutter_riverpod`
- 理由: 画面状態、リポジトリ、非同期更新を整理しやすい
- いまの判断: 採用
- 自前実装との比較:
  - 小さく始めるなら `setState` や簡単な Controller でも可能
  - ただし複数画面と永続化が増えると管理コストが上がりやすい

### 4.3 追加認証

- 分類: `should`
- 代表候補: `local_auth`
- 理由: バックアップ作成、復元、重要設定変更で追加認証が必要になるため
- いまの判断: 採用
- 自前実装との比較:
  - OS 標準の認証 UI を使うほうが安全で現実的
  - 自前で認証 UI を作るのは避けたい

### 4.4 秘密情報の保管

- 分類: `should`
- 代表候補: `flutter_secure_storage`
- 理由: DB 鍵やバックアップ鍵のような小さい秘密情報を OS の安全領域に預けやすい
- いまの判断: 採用
- 自前実装との比較:
  - Keychain / Encrypted Shared Preferences のラッパーを自前で持つより安全に始めやすい
  - ただし保存対象は小さな値に限定する

### 4.5 表示整形

- 分類: `should`
- 代表候補: `intl`
- 理由: 日付、数値、通貨、ロケール処理を揃えやすい
- 自前実装との比較:
  - 自前整形もできるが、ロケール対応で手間が増える

## 5. できれば避けたい候補

### 5.1 重い統合フレームワーク

- 分類: `can avoid`
- 理由: CHOBO はローカルファーストで、まずは UI と永続化境界を薄く保ちたい
- 例:
  - 過剰に多機能なアプリ基盤
  - 解析や計測を前提にした大きな SDK

### 5.2 分析系の追加 SDK

- 分類: `can avoid`
- 理由: MVP は「お金の流れの可視化」が主で、広告や行動分析は不要

### 5.3 OCR / 画像認識 / 地図 / 位置連携

- 分類: `can avoid`
- 理由: MVP 非目的であり、依存と運用コストだけが増えやすい

### 5.4 ネットワーク基盤

- 分類: `can avoid`
- 理由: MVP は同期前提ではなく、外部通信は当面不要

## 6. いまの判断

- `drift`: 採用
- `drift_flutter`: 採用
- `sqlite3_flutter_libs`: 採用
- `pointycastle`: 採用
- `path_provider`: 採用
- `file_picker`: 採用
- `share_plus`: 見送り
- `flutter_secure_storage`: 採用
- `local_auth`: 採用
- `flutter_riverpod`: 採用
- `go_router`: 採用
- `intl`: 採用

## 6. 追加前に詰める質問

新しい依存が出たら、次を必ず確認する。

- 何を解決するのか
- 自前実装だと何がつらいのか
- Android / iOS の両方で成立するか
- 依存が壊れたときの撤退先はあるか
- その依存は CHOBO の中でどの層に閉じ込めるか
- それは MVP に本当に必要か

## 7. いまの仮置き

いまの CHOBO では、次の順で依存を検討するのがよい。

1. Flutter / Dart
2. ローカル DB アクセス層
3. 暗号化と鍵保管
4. バックアップ入出力
5. ルーティング
6. 状態管理
7. 追加認証
8. 表示整形

## 8. 次の議論

この文書を元に、次は以下を決める。

- ローカル DB は `sqflite` 寄りか `drift` 寄りか
- 状態管理は `Riverpod` を採るか、もっと薄く始めるか
- バックアップ入出力はどのプラグイン境界に閉じ込めるか
- iOS 追随を見据えて、最小限に留めるべき依存はどれか
